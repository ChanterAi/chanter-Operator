## 1. Summary

P1.0 adds a real (non-mock) read-only local command runner to CHANTER Operator. The runner executes only 5 pre-approved git read-only commands within a configured workspace. All other commands, injection attempts, and path escapes are blocked. Each execution produces a full audit trail and persists results to the database.

This is **not** autonomous execution, not an agent runner, and not SafeCommit integration. It is a bounded read-only evidence runner accessed through the Operator console.

## 5. Backend Changes

### Architecture
- **New class:** `RealReadonlyRunner` in `src/runners/realReadonlyRunner.ts`
  - Uses `child_process.execFile` — **not** `exec`, **not** `spawn` with shell
  - `shell: false` enforced at the Node.js level
  - `windowsHide: true` to prevent window popup on Windows
  - `timeout` per command (default 15s)
  - `maxBuffer: 1MB` output limit
  - `cwd` set to configured runner workspace
  - Returns `ReadonlyCommandResult` with full metadata

- **Allowlist enforcement (5 commands only):**
  ```
  git status --short
  git diff --stat
  git diff --check
  git show --stat --oneline HEAD
  git show --name-only HEAD
  ```

- **Injection detection** — blocks: `;`, `&&`, `||`, `|`, `>`, `<`, backtick, `$()`, `${`, `&`, `\n`, `\r`

- **No** raw shell strings. Commands parsed as executable + args via whitespace split.

### API Endpoints
| Endpoint | Method | Purpose |
|----------|--------|---------|
| `/api/commands/run` | POST | Execute a read-only command. Body: `{ "command": "git status --short" }` |
| `/api/commands/results` | GET | List recent results. Query: `?limit=50` |

### Audit Events (5 new types)
| Event | When |
|-------|------|
| `readonly_command_requested` | Command submitted for execution |
| `readonly_command_allowed` | Command passed allowlist check |
| `readonly_command_blocked` | Command rejected by policy |
| `readonly_command_completed` | Command executed with exit 0 |
| `readonly_command_failed` | Command executed with non-zero exit |

### Database
- New table `readonly_command_results` stores: id, command, executable, args (JSON), verdict, stdout, stderr, exit_code, duration_ms, workspace_root, timestamp, error
- Indexed by timestamp for fast retrieval

### Configuration
- `OPERATOR_RUNNER_WORKSPACE` env var — path where git commands execute (default: undefined = runner disabled)
- Runner is opt-in: without this env var, `POST /api/commands/run` returns 503

## 7. Audit/Evidence Behavior

Every command execution (allowed or blocked) produces:
1. **Audit event** — recorded to `data/audit.jsonl` via the existing `AuditLogger`
2. **Database row** — persisted to `readonly_command_results` via SQLite transaction
3. **Transaction safety** — if DB write fails, the result is still returned but the error is surfaced

For allowed commands that execute: 3 audit events (requested → allowed → completed/failed)
For blocked commands: 2 audit events (requested → blocked)

## 9. Known Limitations

1. **Runner is opt-in** — requires `OPERATOR_RUNNER_WORKSPACE` env var to be set. Without it, the runner endpoint returns 503.
2. **No task association** — runner commands are independent of the task workflow. They don't create task intents or steps.
3. **Git must be in PATH** — the runner spawns `git` via `execFile`, so git must be on the system PATH.
4. **No streaming output** — stdout/stderr are buffered and returned only after command completion.
5. **No command cancellation** — once started, a command runs until completion or timeout.
6. **Frontend refresh on tab switch** — the runner panel fetches history on mount but doesn't poll for updates.
7. **Workspace validation** — the runner validates the workspace directory exists at policy-check time, but the OS-level `cwd` is enforced by `execFile`. Path validation within git commands is done by git itself (the runner doesn't parse git args for path directives beyond the allowlist match).
